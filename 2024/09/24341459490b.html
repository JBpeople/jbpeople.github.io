1. 基础知识

1.1. 类的常用知识

1.1.1. 私有属性是君子协议

[!warning]
在日常编程中，我们极少使用双下划线来标示一个私有属性。如果你认为某个属性是私有的，直接给它加上单下划线 _ 前缀就够了。而 “标准”的双下划线前缀，反而可能会在子类想要重写父类私有属性时带来不必要的麻烦。

1.1.2. 实例内容都在字典里

类和实例的所有属性都放在一个 __dict__ 当中，我们可以通过直接修改 __dict__ 来改变对象或者属性的值，也可以通过 setattr 来修改，但是两者有所区别。

setattr 修改属性一定会经过里面的校验逻辑，但是通过 __dict__ 修改就可以绕过校验。

1.2. 内置类方法装饰器

1.2.1. 类方法

[!note]
类方法可以通过类的实例调用，也可以通过类调用。

1.2.2. 静态方法

如果函数不会用到实例里面的任何内容就可以改写为静态方法。

[!note]
静态方法可以改写为普通函数，应该可以使用来判断是否改写：

  1. 如果静态方法特别通用，与类的关系不大，改写为普通函数会更好；
  2. 如果静态方法与类关系紧密，则使用静态方法更好；
  3. 相比普通方法，静态方法可以被子类继承，重写；

1.2.3. 属性装饰器

在一个类中，属性表示状态通过 inst.attr 访问，方法代表行为通过 inst.func() 调用。不过装饰器 @property 模糊两者区别，可以把方法通过属性的方式暴露出来。

[!note]
定义 setter 方法会在属性赋值前被调用；
定义 deleter 方法会在属性删除后被调用；

[!warning]
属性对于调用者来说期望很快拿到结果，而方法调用者会愿意等待些许时间去拿到结果。如果方法需要花费几秒钟才能执行完成，那么不建议改成属性。

1.3. 鸭子类型及其局限性

Python 代码是一种鸭子类型的编程语言，在鸭子类型下的编程语言，如果想要操作某个对象不会去判定对象是不是某种类型，而是直接操作，大不了报错，例如读取文件。

代码不会判断 f 是不是一个文件，甚至用户可以传一个数字，一个字符串，无非就是报错。虽然 Python 里面有类型判断 isinstance，但是不建议使用。这样做的优势在于省去类型检查的繁琐工作，不关注类型只关注是否实现了接口。

1.6. 其他知识

1.6.1. Mixin 模式

把某个额外的功能混入类的技术，其他语言对该技术有额外的支持，在 Python 中该技术可以通过多重继承实现。该类的定义名称都是以 Mixin 结尾，大部分该类都不能单独使用，必须继承自别的类。

2. 案例故事

许多人常常把继承当成一种廉价的代码复用手段，当他们看到新需求可以复用某个类的方法时，就会直接创建一个继承该类的子类，快速达到复用目的。但这种简单粗暴的做法忽视了继承的复杂性，容易在未来惹来麻烦。

继承之前问问自己几个问题：

  • 我要让 B 类继承 A 类，但 B 和 A 真的代表同一种东西吗？如果它俩不是同类，为什么要继承？
  • 即使 B 和 A 是同类，那它们真的需要用继承来表明类型关系吗？要知道，Python 是鸭子类型的，你不用继承也能实现多态。
  • 如果继承只是为了让 B 类复用 A 类的几个方法，那么用组合来替代继承会不会更好？

针对事物的行为建模会比针对事物本身建模更具有灵活性。

3. 编程建议

3.1. 使用 __init_subclass__ 替代元类

创建子类的时候会按照元类指定方法。

3.2. 在分支中寻找多态

找到使用多态的时机：

  • 有需要 if/else 的判断，这些判断语句非常类似；
  • 有许多针对类型的 isinstance() 判断；

3.3. 有序组织类方法

类方法的顺序不影响程序的执行结果，但是会影响其他人的阅读体验。

  • 作为惯例，__init__ 实例化方法应该总是放在类的最前面，__new__ 方法同理。
  • 应该把公有方法应该放在类的前面，因为它们是其他模块调用类的入口，是类的门面，也是所有人最关心的内容。以 _ 开头的私有方法，大部分是类自身的实现细节，应该放在靠后的位置。
  • 至于类方法、静态方法和属性对象，你不必将它们区分对待，直接参考公有 / 私有的思路即可。比如，大部分类方法是公有的，所有它们通常会比较靠前。而静态方法常常是内部使用的私有方法，所以常放在靠后的位置。

类方法的组织顺序应该应用于所有类，保持统一。
