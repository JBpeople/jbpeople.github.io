1. DIP：依赖倒置原则

单元测试重要的指导原则：测试程序的行为，而不是测试程序具体实现。意思是应该关心被测试对象的功能是否正常，能否做好自己的事情，而不应该关心被测试对象的内部实现是什么。因为程序的内部实现是经常会变动的，比如今天用 requests 明天发现 fast_requests 会更快就会替换成新的。

使用 mock 编写单元测试只能当做一种应急的技术，如果程序中有太多的 mock 应该考虑程序设计是否合理。

DIP 原则：高层模块不应该依赖于低层模块，二者都应该依赖于抽象。Python 中的抽象类通过 abc.ABC 实现。

如果有一个高层次的类需要通过 requests 请求网页获取内容，就出现了高层模块依赖低层模块。这个时候需要创建一个中间抽象，让高层模块依赖抽象，低层模块依赖上方的抽象层，这就是倒置。

1.1. DIP 小结

ABC 不是严格的抽象方案，在Python 3.8 之后在 typing 中出现了更加严格的方案 protocol 。

非大型项目一般 ABC 足够满足正常使用需求了。

抽象不是越多越好，根据使用情况来决定，这涉及到后面的 ISP 原则。

2. ISP：接口隔离原则

根据高层模块的需求对抽象类进行分解，比如高层模块 A 需要抽象类 C 的 a 方法，高层模块 B 需要抽象类 C 的 a, b, c 方法，这种情况不应该把所有的方法都放在 C 下面，应该把抽象类 C 分割成两个。

设计原则，接口给上层提供的方法，一定是一个不多一个不少。

2.1. 其他违反ISP 场景

可以看到为了判断 COOKIES 是否满足条件，我们导入了整个 requests 模块。这个就是低层模块提供的方法超过了调用者的需求，我们完全不需要 requests 的其他任何内容。因此完全可以把函数改成接收字典数据。

3. 总结

SRP：单一职责原则

  • 一个类只应该有一种被修改的原因
  • 编写更小的类通常更不容易违反 SRP
  • SRP 同样适用于函数，你可以让函数和类协同工作

OCP：开放-关闭原则

  • 类应该对修改关闭，对扩展开放
  • 通过分析需求，找到代码中易变的部分，是让类符合 OCP 的关键
  • 使用子类继承的方式可以让类符合 OCP
  • 通过算法类与依赖注入，也可以让类符合 OCP
  • 将数据与逻辑分离，使用数据驱动的方式也是实践 OCP 的好办法

LSP：里氏替换原则

  • LSP 认为子类应该可以任意替代父类使用
  • 子类不应该抛出父类不认识的异常
  • 子类方法应该返回与父类一致的类型，或者返回父类返回值的子类型对象
  • 子类的方法参数应该和父类方法完全一致，或者要求更为宽松
  • 某些类可能会存在隐式合约，违反这些合约也会导致违反 LSP

DIP：依赖倒置原则

  • DIP 认为高层模块和低层模块都应该依赖于抽象
  • 编写单元测试有一个原则：测试行为，而不是测试实现
  • 单元测试不宜使用太多 mock，否则需要调整设计
  • 依赖抽象的好处是，修改低层模块实现不会影响高层代码
  • 在 Python 中，你可以用 abc 模块来定义抽象类
  • 除 abc 以外，你也可以用 Protocol 等技术来完成依赖倒置

ISP：接口隔离原则

  • ISP 认为客户依赖的接口不应该包含任何它不需要的方法
  • 设计接口就是设计抽象
  • 写更小的类、更小的接口在大多数情况下是个好主意
